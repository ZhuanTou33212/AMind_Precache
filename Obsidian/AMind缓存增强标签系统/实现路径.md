# AMind 缓存增强标签系统 - 实现路径

## 目标

在现有“本地缓存 2000 张图片”的基础上，为每张缓存图片增加本地标签能力。用户可以通过快捷键或鼠标选择标签，例如：

- 惊艳
- 好看
- 一般
- 不堪
- 带水印

标签可以组合，例如：

- 好看_带水印
- 不堪_带水印

之后插件根据本地标签自动在 AMiner 网页中选择对应选项，并按批次自动提交。

## 推荐实现路线

第一版采用“本地标签 + DOM 自动选择 + 快捷键提交”的方案。

原因：

- 当前扩展已经运行在 AMiner 页面内，可以读取 DOM、监听键盘、触发按钮点击。
- 当前 `monitor-main.js` 已经能拦截 XHR/fetch，可继续用它观察提交接口，但不必一开始就绕过页面直接调用云端。
- 直接调用 AMiner 提交接口需要完整复刻请求体、Token、题目 ID、选项 ID、确认流程和后端校验，风险更高。
- 使用页面已有按钮提交，能最大程度复用网页自己的校验和确认逻辑。

第二版再考虑“直接 API 提交”。前提是已经稳定抓到 AMiner 提交接口的请求体，并能在测试账号上验证重复提交、失败重试、确认弹窗绕过后的状态一致性。

## 模块拆分

### 1. 本地标签配置

位置：桌面程序 UI。

新增“标签配置”面板，用户可以配置标签组：

```json
{
  "groups": [
    {
      "id": "quality",
      "name": "图片质量",
      "mode": "single",
      "options": [
        { "id": "amazing", "label": "惊艳", "hotkey": "Q", "aliases": ["惊艳"] },
        { "id": "good", "label": "好看", "hotkey": "W", "aliases": ["好看"] },
        { "id": "normal", "label": "一般", "hotkey": "E", "aliases": ["一般"] }
      ]
    },
    {
      "id": "watermark",
      "name": "水印",
      "mode": "multi",
      "options": [
        { "id": "watermark", "label": "带水印", "hotkey": "A", "aliases": ["带水印", "水印", "logo"] }
      ]
    }
  ],
  "submit": {
    "enabled": true,
    "batchSize": 5,
    "strategy": "keyboard"
  }
}
```

说明：

- `mode=single`：同组只能选一个，例如“惊艳/好看/一般”。
- `mode=multi`：同组可多选，例如“带水印/文字遮挡”等。
- `aliases`：用于匹配网页选项文字，避免网页文案和本地标签完全不一致。

### 2. 本地缓存标签存储

位置：Go 本地服务 + SQLite。

在 `images` 表中增加字段：

```sql
labels TEXT DEFAULT ''
label_status TEXT DEFAULT 'pending'
submitted_at INTEGER DEFAULT 0
```

推荐存储 JSON 字符串，而不是只存 `好看_带水印`。展示时再拼接后缀名。

```json
{
  "quality": ["好看"],
  "watermark": ["带水印"]
}
```

这样后续映射网页选项更稳定。

### 3. 快捷键与鼠标标注

位置：新增扩展脚本 `monitor-quicklabel.js`。

职责：

- 从 `http://127.0.0.1:9800/api/labels-config` 读取标签配置。
- 监听 `keydown`，忽略输入框、文本框、可编辑区域。
- 根据当前题号找到缓存图片记录。
- 更新该图片的本地标签。
- 在页面中自动选择对应选项。
- 必要时显示一个小浮层，提示当前图片已选标签。

快捷键规则：

- 单选组：按下新标签时替换同组旧标签。
- 多选组：按下同一标签时切换选中/取消。
- `Backspace` 或自定义清空键：清空当前题标签。
- `Enter` 或自定义提交键：提交当前题。

### 4. 网页选项识别

位置：`monitor-quicklabel.js`，运行在页面 DOM 环境。

第一版不从 HTTP 层解析按钮，而是从 DOM 层识别选项：

1. 读取页面中可点击元素、label、radio、checkbox、button。
2. 用 `innerText`、`aria-label`、`title`、相邻文本做匹配。
3. 用标签配置中的 `label + aliases` 匹配网页选项文案。
4. 命中后触发 `click()`。

推荐保留一个调试接口：

```js
window.__aminerLabelDebug.scanOptions()
```

返回当前页面识别到的选项，方便调网页结构变化。

### 5. 自动提交

第一版推荐使用键盘/按钮模拟提交：

```text
选择选项 -> Ctrl+Enter -> 等待确认弹窗 -> Enter
```

或：

```text
选择选项 -> 找到“提交”按钮并 click -> 找到确认按钮并 click
```

优先级：

1. DOM 点击提交按钮和确认按钮。
2. 找不到按钮时 fallback 到 `Ctrl+Enter` + `Enter`。
3. 直接 AMiner API 提交放到第二版。

理由：

- 页面已有逻辑会处理必填校验、选项 ID、题目状态、确认弹窗。
- 直接 API 提交需要逆向请求体，且网页结构/接口变化后更容易静默失败。
- 对标注任务来说，错误提交比慢一点更糟。

### 6. 批量自动提交

当本地缓存中连续 N 张图片都已有完整标签时，扩展进入自动处理队列：

```text
取当前题号 -> 找本地标签 -> 选网页选项 -> 提交 -> 等页面跳下一题 -> 继续
```

第一版建议默认 `batchSize=5`，且必须有暂停机制：

- `Esc`：立即暂停自动提交。
- 提交失败 1 次：停止队列并提示。
- 页面题号和缓存题号不一致：停止队列。
- 找不到对应网页选项：停止队列。

## 数据流

```text
本地服务预缓存图片
  -> SQLite 保存 hash/url/promptId/questionNum
  -> 扩展将页面 OSS 图片替换为本地图片
  -> 用户按快捷键或点击标签
  -> 扩展 POST /api/image-labels
  -> 本地服务更新 SQLite labels
  -> 扩展根据 labels 点击网页选项
  -> 扩展触发提交
  -> 本地服务记录 submitted 状态
```

## 阶段拆分

### 阶段 1：只做本地标签

目标：

- UI 可配置标签。
- 当前题可通过快捷键写入本地标签。
- `/api/images` 能看到 labels。
- 不自动点击网页，不自动提交。

验收：

- 缓存 5 张图片。
- 给第 1、2、3 张打不同标签。
- 重启程序后标签仍存在。

### 阶段 2：自动选择网页选项

目标：

- 根据本地标签找到网页选项。
- 自动点击 radio/checkbox。
- 不自动提交。

验收：

- `好看_带水印` 能选中“好看”和“带水印”两个网页项。
- 找不到选项时给出可见提示，不提交。

### 阶段 3：半自动提交

目标：

- 当前题标签完整后，按一个快捷键完成“选择 + 提交 + 确认”。

验收：

- 单题提交后，AMiner 页面进入下一题。
- 本地状态变为 submitted。

### 阶段 4：批量自动提交

目标：

- 已标完的连续 N 张图自动提交。

验收：

- 连续 5 张图已打标签时，自动提交 5 张。
- 任意一题选项匹配失败时立即停止。

### 阶段 5：研究直接 API 提交

目标：

- 仅作为优化项，不作为第一版主路径。

验收：

- 捕获并记录提交接口 URL、method、headers、body。
- 能用测试账号复现一次直接提交。
- 对失败状态有明确回滚或停止策略。

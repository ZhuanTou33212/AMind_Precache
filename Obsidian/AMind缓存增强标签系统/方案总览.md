# AMind 缓存增强标签系统 - 方案总览

## 结论

可以实现。

当前插件已经能缓存大量图片，并把网页 OSS 图片替换成本地缓存图片。下一步可以在本地缓存记录上增加标签字段，让用户用快捷键或鼠标给图片打标签，再由扩展把本地标签映射到 AMiner 网页选项，最后自动提交。

第一版建议走：

```text
本地标签 -> 网页 DOM 自动选项 -> 页面提交按钮/快捷键提交
```

不建议第一版直接绕过网页调用 AMiner 云端提交接口。

## 对需求的拆解

### 1. 给本地缓存图片添加后缀标签

实现方式：

- 不直接改图片文件名。
- 在 SQLite 的图片记录里增加 `labels` 字段。
- UI 展示时把标签拼成后缀，例如 `好看_带水印`。

原因：

- 改文件名会影响 `/api/image/{hash}` 的读取。
- JSON 标签更适合多选和后续自动映射。
- 仍然可以在导出时生成带后缀的文件名。

### 2. 快捷键或鼠标选择标签

实现方式：

- 桌面 UI 负责配置标签组和快捷键。
- 扩展脚本监听网页快捷键。
- 快捷键命中后写入本地服务。
- 鼠标模式可以用页面浮层或桌面 UI 选择。

标签组支持：

- 单选：惊艳、好看、一般、不堪。
- 多选：带水印、文字遮挡、低清晰度等。

组合结果：

```text
好看_带水印
不堪_带水印
```

### 3. 根据本地标签选择网页按钮

实现方式：

- 扩展读取当前题号。
- 用题号找到本地缓存图片的 labels。
- 在网页 DOM 中扫描 radio、checkbox、button、label。
- 用标签名和别名匹配网页文案。
- 自动 click 对应选项。

这里不需要第一版就“抓取 HTTP 层中的按钮”。按钮属于 DOM 层内容，直接扫描 DOM 更容易实现。HTTP 层更适合用来观察提交请求，而不是识别页面按钮。

### 4. 自动提交

有两个方案。

#### 方案 A：模拟页面提交

方式：

```text
选中网页选项 -> 点击提交按钮 -> 点击确认按钮
```

或者：

```text
Ctrl+Enter -> 等待确认弹窗 -> Enter
```

推荐作为第一版。

原因：

- 实现更快。
- 复用网页自己的校验逻辑。
- 不需要逆向提交接口。
- 出错时用户能在页面上看到状态。

#### 方案 B：直接提交到云端 API

方式：

```text
扩展拦截提交请求 -> 解析请求体 -> 本地服务直接 POST AMiner API
```

不建议第一版做。

原因：

- 需要完整确认请求体结构。
- 需要知道网页选项对应的后端 ID。
- 后端可能校验确认流程、题目状态、重复提交。
- 一旦映射错，可能批量错误提交。

适合作为第二版优化。

## 推荐里程碑

### M1：本地标签可用

用户能配置标签，并用快捷键给缓存图片写入标签。

### M2：网页选项自动选择

插件能根据 `好看_带水印` 自动选中网页上的“好看”和“带水印”。

### M3：单题自动提交

用户按一个快捷键，完成当前题的选择、提交、确认。

### M4：五题批量自动提交

本地连续 5 张缓存图片都有完整标签时，自动提交 5 题。失败立即暂停。

### M5：研究直接 API 提交

仅在确认提交接口稳定后实现。

## 风险点

### 页面 DOM 变化

AMiner 页面改版后，选项识别可能失效。

应对：

- 保留 `scanOptions()` 调试函数。
- 用文本和别名匹配，不写死单一选择器。

### 快捷键误触

用户输入文本时不应触发标签。

应对：

- 忽略 input、textarea、contenteditable。
- 批量提交必须有明确快捷键。
- `Esc` 随时暂停。

### 标签和网页选项不一致

本地标签“带水印”可能对应网页文案“带有水印/logo等非画面内自然文字”。

应对：

- 标签配置中增加 `aliases`。
- UI 允许用户配置网页别名。

### 批量错误提交

这是最大风险。

应对：

- 第一版不直接调用云端提交接口。
- 每题提交前检查题号一致。
- 找不到选项就停止。
- 提交后必须确认页面进入下一题。

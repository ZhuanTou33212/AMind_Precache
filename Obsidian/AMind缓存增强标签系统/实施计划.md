# 本地缓存标签系统 - 实施计划

## 当前基础

现有插件已经具备这些能力：

- 本地服务监听 `127.0.0.1:9800`。
- SQLite 存储缓存图片：`hash/url/promptId/questionNum/reason/cachedAt/fileSize`。
- 扩展从 AMiner 页面读取当前题号。
- 扩展把 OSS 图片替换成本地缓存图片：`/api/image/{hash}`。
- 扩展已能读取 Token、taskId、startDate，并同步到本地服务。

增强标签系统应当复用这些基础，不重新做一套缓存链路。

## 一期目标

实现“本地标签 -> 网页选项 -> 自动提交”的稳定闭环：

```text
缓存图片 -> 本地打标签 -> 页面自动选项 -> 页面提交 -> 本地记录状态
```

第一版不直接绕过 AMiner 页面提交接口。

## 数据库设计

### images 表新增字段

```sql
ALTER TABLE images ADD COLUMN labels TEXT DEFAULT '';
ALTER TABLE images ADD COLUMN label_status TEXT DEFAULT 'pending';
ALTER TABLE images ADD COLUMN submitted_at INTEGER DEFAULT 0;
```

字段含义：

| 字段 | 含义 |
|---|---|
| `labels` | JSON 字符串，记录每个标签组的选择 |
| `label_status` | `pending/labeled/submitting/submitted/error` |
| `submitted_at` | 成功提交时间戳，未提交为 0 |

### labels 示例

```json
{
  "quality": ["好看"],
  "watermark": ["带水印"]
}
```

展示给用户时再拼接为：

```text
好看_带水印
```

## API 设计

### GET /api/labels-config

扩展读取标签配置。

返回：

```json
{
  "groups": [],
  "submit": {}
}
```

### POST /api/labels-config

桌面 UI 保存标签配置。

请求体同上。

### POST /api/image-labels

写入单张图片标签。

请求：

```json
{
  "questionNum": 1,
  "promptId": "xxx",
  "hash": "optional",
  "labels": {
    "quality": ["好看"],
    "watermark": ["带水印"]
  }
}
```

匹配优先级：

1. `hash`
2. `promptId`
3. `questionNum`

### POST /api/image-label-status

更新提交状态。

请求：

```json
{
  "questionNum": 1,
  "status": "submitted",
  "message": ""
}
```

### GET /api/images

现有接口扩展返回字段：

```json
{
  "hash": "...",
  "url": "...",
  "promptId": "...",
  "questionNum": 1,
  "labels": {
    "quality": ["好看"],
    "watermark": ["带水印"]
  },
  "labelText": "好看_带水印",
  "labelStatus": "labeled",
  "submittedAt": 0
}
```

## 扩展脚本设计

新增：

```text
extension/monitor-quicklabel.js
```

运行环境建议：`ISOLATED world` 即可。

它负责：

1. 读取 `/api/labels-config`。
2. 读取 `/api/images`，建立 `questionNum -> imageRecord` 映射。
3. 监听快捷键。
4. 按标签组规则更新当前题标签。
5. 调用 `/api/image-labels` 保存标签。
6. 根据标签点击网页选项。
7. 在用户确认或自动模式开启时执行提交。

## 快捷键规则

### 基础行为

- 输入框、文本框、contenteditable 中不响应快捷键。
- 单选组：按下对应键后替换同组旧标签。
- 多选组：按下对应键后切换标签。
- `Esc`：暂停自动提交队列。
- `Ctrl+Enter`：保留页面原有提交快捷键。

### 推荐新增快捷键

| 快捷键 | 行为 |
|---|---|
| `Q/W/E/R` | 图片质量单选 |
| `A/S/D...` | 附加标签多选 |
| `Backspace` | 清空当前题本地标签 |
| `Alt+Enter` | 按本地标签选择网页选项并提交当前题 |
| `Alt+B` | 提交已标好的连续 batch |

## 网页选项识别策略

不要第一版就从 HTTP 请求中反推按钮。第一版优先从 DOM 识别，因为页面里已经有可点击控件。

识别顺序：

1. `label` 文本匹配。
2. `button` 文本匹配。
3. `input[type=radio|checkbox]` 的相邻文本匹配。
4. `aria-label/title` 匹配。
5. React/Vue 组件外层文本匹配后 click。

匹配文本来源：

- 标签 `label`
- 标签 `aliases`
- 去空格、去标点后的文本

调试函数：

```js
window.__aminerQuickLabel.scanOptions()
window.__aminerQuickLabel.applyLabels({ quality: ["好看"], watermark: ["带水印"] })
```

## 提交策略判断

### 方案 A：模拟页面提交

流程：

```text
点击网页选项 -> 点击提交按钮 -> 点击确认按钮
```

找不到按钮时 fallback：

```text
Ctrl+Enter -> 等待确认弹窗 -> Enter
```

优点：

- 实现最快。
- 复用网页校验。
- 后端接口变化时更稳。
- 不需要逆向提交请求体。

缺点：

- 依赖页面 DOM 和快捷键。
- 页面改版时需要更新选择器。

### 方案 B：直接上传到云端服务器

流程：

```text
抓取提交接口 -> 复刻请求体 -> 直接 POST AMiner API
```

优点：

- 理论上最快。
- 不依赖按钮和弹窗。

缺点：

- 需要确认选项 ID、题目 ID、任务 ID、提交体结构。
- 可能绕过前端校验，错误提交风险高。
- 接口变化时容易静默失败。
- 需要处理重复提交、失败重试、后端状态同步。

结论：

第一版选方案 A。方案 B 放到二期研究，只能在抓到稳定请求体并完成测试账号验证后启用。

## 自动批量提交

自动批量提交只处理“连续且完整”的题。

完整定义：

- 当前题在本地缓存中存在。
- 必填标签组都已选择。
- 对应网页选项都能找到。
- 页面当前题号与缓存 `questionNum` 一致。

流程：

```text
读取当前题号
  -> 查本地标签
  -> 校验必填组
  -> 点击网页选项
  -> 提交
  -> 等待题号变化
  -> 继续下一题
```

停止条件：

- 用户按 `Esc`。
- 找不到网页选项。
- 提交后题号没有变化。
- AMiner 返回错误。
- 当前题没有完整标签。

## 验收步骤

### 1. 数据库迁移

验证：

- 旧 `cache.db` 能启动。
- 新字段自动补齐。
- `/api/images` 返回 `labels` 字段。

### 2. 标签配置 UI

验证：

- 新建标签组。
- 添加标签。
- 保存后重启程序仍存在。

### 3. 快捷键本地标注

验证：

- 第 1 题按 `W` 后，本地显示 `好看`。
- 再按 `A` 后，本地显示 `好看_带水印`。
- 重启后标签不丢失。

### 4. 网页选项点击

验证：

- `好看_带水印` 能在网页中选中两个对应项。
- 选项不存在时停止，并显示失败原因。

### 5. 单题提交

验证：

- `Alt+Enter` 完成当前题选择和提交。
- 本地状态变为 `submitted`。
- AMiner 页面进入下一题。

### 6. 五题批量提交

验证：

- 本地连续 5 张图片已打完整标签。
- `Alt+B` 后自动提交 5 题。
- 任意一题失败时立即停止，不跳过失败题。
